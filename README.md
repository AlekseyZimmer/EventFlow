# EventFlow (ТаймМастер) ⏱️👑

EventFlow — это полностековое веб-приложение для управления расписанием мероприятий в реальном времени. Оно позволяет организаторам создавать события, составлять поминутное расписание (вручную или через импорт Excel), управлять ходом мероприятия (запускать и завершать этапы), а участникам — следить за актуальным статусом события с любого устройства без задержек.

## 🌟 Ключевые возможности

* **Синхронизация в реальном времени:** Использование WebSockets (Socket.IO) для мгновенного обновления расписания у всех участников при любых изменениях со стороны организатора.
* **Умное расписание:** Автоматический расчет времени начала следующего этапа на основе окончания предыдущего.
* **Drag-and-Drop:** Удобная сортировка этапов перетаскиванием (с мгновенной синхронизацией).
* **Импорт и Экспорт Excel:** Скачивание шаблона расписания и массовая загрузка этапов из `.xlsx` файлов.
* **Разделение ролей:** 
  * *Организатор* (создатель) имеет полный доступ к управлению (защищено уникальным токеном в браузере).
  * *Участник* (зритель) может присоединиться по коду (ID) и наблюдать за ходом мероприятия.
* **Мультиязычность:** Встроенная поддержка английского (EN), русского (RU) и шуточного языка эмодзи (EMJ).
* **Геолокация:** Интеграция с API Яндекс.Карт для подсказок при вводе места проведения.
* **Автоматическая очистка БД:** Настраиваемая политика хранения данных (1, 7, 14 или 30 дней после мероприятия), после чего данные безвозвратно удаляются из SQLite.

## 📋 Системные требования (Prerequisites)

Для успешного запуска приложения на собственном сервере (VDS/VPS или локальной машине) требуются следующие версии ПО:

* **Node.js:** `v18.18.0` или выше (рекомендуется `v20.x LTS`)
* **npm:** `v9.0.0` или выше
* **ОС:** Linux (Ubuntu/Debian), macOS или Windows.

*Примечание: Пакет `better-sqlite3` компилирует нативные бинарные файлы (C++), поэтому в системе должны быть установлены базовые инструменты сборки (например, `build-essential` и `python3` для Linux, или "Desktop development with C++" для Windows).*

## 🚀 Установка и запуск

### 1. Клонирование репозитория
```bash
git clone <url-вашего-репозитория>
cd <папка-проекта>
```

### 2. Установка зависимостей
```bash
npm install
```

### 3. Настройка переменных окружения
Создайте файл `.env` в корне проекта (рядом с `package.json`) и добавьте необходимые ключи:

```env
# API-ключ Яндекс.Карт (Геокодер) для подсказок адресов
VITE_YANDEX_API_KEY=ваш_ключ_здесь
```
*(Получить бесплатный ключ можно в [Кабинете разработчика Яндекса](https://developer.tech.yandex.ru/))*

### 4. Запуск в режиме разработки (Development)
```bash
npm run dev
```
Приложение будет доступно по адресу: `http://localhost:3000`

### 5. Сборка и запуск в Production
Для запуска на боевом сервере необходимо сначала собрать Frontend, а затем запустить серверный скрипт:

```bash
# 1. Сборка клиентской части (Vite) в папку /dist
npm run build

# 2. Запуск production-сервера
npm start
```
*В production-режиме Express автоматически раздает статические файлы из папки `dist`.*

## 🗄 База данных (SQLite)

Приложение использует легковесную БД SQLite. Файл базы данных `events.db` создается автоматически в корне проекта при первом запуске сервера.

**Таблицы:**
1. `events`: Хранит ID, название, дату, локацию, статус, токен владельца и срок хранения (retentionDays).
2. `stages`: Хранит этапы расписания, привязанные к `eventId` (связь Foreign Key с каскадным удалением).

**Очистка данных:**
В `server.ts` работает фоновый интервал (`setInterval`), который каждый час проверяет БД и удаляет мероприятия, срок хранения которых истек (Дата мероприятия + `retentionDays` < Текущая дата).

## 🔌 WebSocket События (Socket.IO)

* `join_event` (Client -> Server): Подключение клиента к комнате конкретного мероприятия (по ID).
* `schedule_updated` (Server -> Client): Сигнал о том, что расписание изменилось (добавлен этап, изменено время, изменен порядок). Клиент запрашивает свежие данные по REST API.
* `event_updated` (Server -> Client): Сигнал об изменении глобального статуса мероприятия (например, переход из `planned` в `active`).
* `stage_updated` (Server -> Client): Сигнал об изменении статуса конкретного этапа (начался/завершился), используется для показа всплывающих уведомлений (Toast) у участников.

## 🛡 Безопасность

* **Авторизация организатора:** При создании мероприятия генерируется `ownerToken` (UUID v4). Он сохраняется в `localStorage` браузера создателя и передается в заголовке `x-owner-token` при любых REST API запросах на изменение (POST, PATCH, DELETE).
* **Защита от SQL-инъекций:** Использование параметризованных запросов (`@param` и `?`) в `better-sqlite3`.
